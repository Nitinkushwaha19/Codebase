

# Linux Kernel Namespaces and Control Groups - Short Notes

## 1. CONTAINER ISOLATION FUNDAMENTALS

### Traditional VMs vs Containers
- **VMs:** Full virtualization with separate OS kernels
- **Containers:** Share host OS kernel with isolation boundaries

**Container Isolation Components:**
- **Namespaces** - Limit what you can see
- **Cgroups** - Limit how much you can use  
- **chroot** - Filesystem isolation

---

## 2. NAMESPACES

### Definition
Namespaces provide process-level isolation by limiting what a process can see and access.

**Formula:** `Namespace = limits what you can see (and therefore use)`

### Types of Namespaces
| Namespace | Purpose | Example |
|-----------|---------|---------|
| **PID** | Process isolation | Container sees only its processes |
| **NET** | Network isolation | Separate network interfaces |
| **MNT** | Mount points | Separate filesystem view |
| **UTS** | Hostname/domain | Container has own hostname |
| **IPC** | Inter-process communication | Separate message queues |
| **USER** | User/group IDs | Map container users to host |

### PID Namespace Example
```bash
# Host process
sleep 1000 &  # PID 19005 on host

# Container with host PID namespace
docker run --pid=host ubuntu ps aux
# Can see and kill host process 19005

# Container with isolated PID namespace  
docker run ubuntu ps aux
# Only sees container processes, PID 1 is container init
```

## 3. CONTROL GROUPS (CGROUPS)

### Definition
Cgroups control and limit resource usage (CPU, memory, I/O) for containers.

**Formula:** `Cgroups = limits how much you can use`

### Docker Cgroup Types
- **CPU groups** - CPU time allocation
- **Memory cgroups** - RAM usage limits
- **BlkIO cgroup** - Disk I/O throttling
- **Device cgroups** - Device access control
- **Network cgroups** - Network bandwidth
- **Freezer cgroups** - Pause/resume processes
- **CPUset cgroups** - CPU core assignment
- **PID cgroups** - Process count limits

### Memory Limits
```bash
# Limit container to 512MB RAM
docker run -m 512m nginx

# Limit with swap
docker run -m 512m --memory-swap 1g nginx

# No swap allowed
docker run -m 512m --memory-swap 512m nginx
```

### CPU
 Limits
```bash
# CPU shares (relative weight)
docker run --cpu-shares 512 nginx

# Limit to specific CPU cores
docker run --cpus="1.5" nginx

# Assign to specific CPU cores
docker run --cpuset-cpus="0,1" nginx

# CPU period and quota
docker run --cpu-period=100000 --cpu-quota=50000 nginx
```

## 4. DOCKER IN DOCKER (DIND)

### Definition
Running Docker containers inside Docker containers, commonly used in CI/CD pipelines.

### Why DinD is Necessary
- **CI/CD pipelines** need to build Docker images
- **Jenkins agents** running as containers need Docker commands
- **Nested containerization** for complex workflows

### How DinD Works
```bash
# Run DinD container
docker run -d \
  --privileged \
  --name dind \
  -v /var/lib/docker \
  docker:dind

# Use DinD from another container
docker run --rm \
  --link dind:docker \
  -e DOCKER_HOST=tcp://docker:2375 \
  docker:latest docker ps
```

### Architecture
```
Host
├── Docker Daemon
├── Container 1
├── Container 2
└── DinD Container
    ├── Docker Daemon (inside)
    ├── Container 3
    └── Container 4
```

## 5. DOCKER REMOTE API

### Definition
Access Docker Engine functionality through REST API for automation and remote management.

### Socket Types
| Type | Path/Address | Use Case |
|------|-------------|----------|
| **UNIX** | `/var/run/docker.sock` | Local access (default) |
| **TCP** | `tcp://0.0.0.0:2375` | Remote access (unencrypted) |
| **TCP+TLS** | `tcp://0.0.0.0:2376` | Remote access (encrypted) |

### Configuration
**daemon.json:**
```json
{
  "hosts": [
    "unix:///var/run/docker.sock",
    "tcp://0.0.0.0:2375"
  ]
}
```

### Remote Access Examples
```bash
# Set remote Docker host
export DOCKER_HOST=tcp://192.168.1.100:2375

# Remote Docker commands
docker -H tcp://192.168.1.100:2375 ps
docker -H tcp://192.168.1.100:2375 info

# API calls
curl http://192.168.1.100:2375/info
curl http://192.168.1.100:2375/containers/json
```

## 6. CONTAINER MONITORING

### Key Metrics to Monitor
- **Image count/size** - Storage usage
- **Container count/size** - Resource usage by state
- **Volume count/size** - Persistent storage
- **CPU usage** - Host CPU utilization
- **Memory usage** - RAM consumption
- **Network bandwidth** - I/O throughput
- **Disk throughput** - Storage I/O
- **Process count** - Running processes

### Monitoring Tools

#### cAdvisor
```yaml
# docker-compose.yml
version: '3'
services:
  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    ports:
      - "8080:8080"
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
```

#### Prometheus
- Collects metrics from cAdvisor
- Time-series database
- Alerting capabilities
- Integrates with Grafana for visualization

### Monitoring Commands
```bash
# System information
docker system info
docker system df          # Disk usage
docker system df -v
       # Detailed disk usage

# Container stats
docker stats              # Live resource usage
docker ps -s             # Container sizes
docker ps -f status=exited # Exited containers

# Cleanup commands
docker system prune       # Remove unused data
docker image prune        # Remove unused images
docker volume prune       # Remove unused volumes
```

## 7. CONTAINER SECURITY

### Common Vulnerabilities
- **Base image vulnerabilities** - Outdated packages
- **Application dependencies** - Vulnerable libraries
- **Configuration issues** - Insecure settings
- **Runtime vulnerabilities** - Container breakouts
- **Supply chain attacks** - Compromised images

### Security Scanning Tools
| Tool | Type | Purpose |
|------|------|---------|
| **Anchore Engine** | Open source | Policy-based scanning |
| **Clair** | Open source | Static vulnerability analysis |
| **Vuls.io** | Open source | Vulnerability scanner |
| **OpenSCAP** | Open source | Security compliance |

### Security Pipeline Example
```bash
# 1. Build image
docker build -t myapp:latest .

# 2. Scan for vulnerabilities
anchore-cli image add myapp:latest
anchore-cli image wait myapp:latest
anchore-cli evaluate check myapp:latest

# 3. Push only if scan passes
if [ $? -eq 0 ]; then
  docker push myapp:latest
fi
```

## 8. CLOUD-NATIVE APPLICATIONS

### Definition
Applications engineered to run on platforms, designed for **resiliency**, **agility**, and **operability**.

### Key Characteristics
- **Resiliency** - Design for failure, graceful degradation
- **Agility** - Rapid deployment and scaling
- **Operability** - Self-monitoring and healing

### Implementation Patterns
- **Microservices** - Decomposed, independent services
- **Health reporting** - Built-in health checks
- **Telemetry** - Metrics, logs, traces
- **Declarative** - Desired state configuration

### Service Models
- **CaaS** - Container as a Service
- **PaaS** - Platform as a Service  
- **FaaS** - Function as a Service

### 12-Factor App Principles
Key principles for building cloud-native applications:
1. **Codebase** - One codebase, many deploys
2. **Dependencies** - Explicitly declare dependencies
3. **Config** - Store config in environment
4. **Backing services** - Treat as attached resources
5. **Build/Release/Run** - Separate stages
6. **Processes** - Stateless processes
7. **Port binding** - Export services via ports
8. **Concurrency** - Scale via process model
9. **Disposability** - Fast startup/shutdown
10. **Dev/Prod parity** - Keep environments similar
11. **Logs** - Treat as event streams
12. **Admin processes** - Run as one-off processes
