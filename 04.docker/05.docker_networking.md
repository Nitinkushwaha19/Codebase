## 1. WHAT IS DOCKER NETWORKING?

**Definition:** Docker networking enables communication between containers, external connectivity, and service discovery using pluggable network drivers.

**Why Important:**
- Container-to-container communication
- External connectivity (internet access)
- Service discovery (find containers by name)
- Load balancing across instances
- Network isolation and security

---

## 2. NETWORK DRIVERS

### Bridge (Default)
**Definition:** Creates private internal network on host with NAT for external access

**Use Cases:** Single-host deployments, development
**IP Range:** Usually 172.17.x.x

```bash
# Default bridge
docker run -d nginx

# Custom bridge
docker network create my-bridge
docker run -d --network my-bridge nginx
```

### Host
**Definition:** Container shares host's network stack directly

**Use Cases:** High-performance networking, legacy apps
**IP:** Uses host's IP address

```bash
docker run -d --network host nginx
# Container accessible on host's IP directly
```

### Overlay
**Definition:** Multi-host networking for distributed containers

**Use Cases:** Docker Swarm, microservices across hosts
**Requirement:** Swarm mode

```bash
# Requires swarm mode
docker swarm init
docker network create -d overlay my-overlay
```

### Macvlan
**Definition:** Assigns MAC address to container, appears as physical device

**Use Cases:** Legacy apps, network monitoring tools

```bash
docker network create -d macvlan \
  --subnet=192.168.1.0/24 \
  --gateway=192.168.1.1 \
  -o parent=eth0 \
  my-macvlan
```

### None
**Definition:** Disables all networking

**Use Cases:** Security isolation, batch processing

```bash
docker run -d --network none alpine
# No network access at all
```

## 3. DEFAULT NETWORKS

Every Docker installation has:
- **bridge** - Default container network
- **host** - Host machine network
- **none** - No networking

```bash
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# xxxxxxxxxxxx   bridge    bridge    local
# xxxxxxxxxxxx   host      host      local  
# xxxxxxxxxxxx   none      null      local
```

## 4. USER-DEFINED NETWORKS

### Why Better Than Default Bridge?
- ✅ **Automatic DNS resolution** (containers reach each other by name)
- ✅ **Better isolation** (only connected containers communicate)
- ✅ **Dynamic connections** (connect/disconnect while running)
- ✅ **Configurable** (custom IP ranges, gateways)

### Creating Custom Networks
```bash
# Basic custom network
docker network create webapp-network

# Advanced configuration
docker network create \
  --driver bridge \
  --subnet=172.20.0.0/16 \
  --gateway=172.20.0.1 \
  my-custom-network
```

## 5. COMMON COMMANDS

```bash
# List networks
docker network ls

# Create network
docker network create <network_name>

# Remove network
docker network rm <network_name>

# Inspect network details
docker network inspect <network_name>

# Connect container to network
docker network connect <network> <container>

# Disconnect container from network
docker network disconnect <network> <container>

# Run container on specific network
docker run --network <network_name> <image>
```

## 6. PRACTICAL EXAMPLES

### Web App with Database
```bash
# Create custom network
docker network create webapp-net

# Run database
docker run -d \
  --name database
 \
  --network webapp-net \
  -e POSTGRES_PASSWORD=secret \
  postgres

# Run web app (can connect to 'database' by hostname)
docker run -d \
  --name frontend \
  --network webapp-net \
  -p 8080:80 \
  nginx
```

### Load Balancer Setup
```bash
# Create network
docker network create lb-network

# Run multiple web servers
docker run -d --name web1 --network lb-network nginx
docker run -d --name web2 --network lb-network nginx
docker run -d --name web3 --network lb-network nginx

# Run load balancer
docker run -d \
  --name loadbalancer \
  --network lb-network \
  -p 80:80 \
  haproxy
```

### Multi-Network Container
```bash
# Create two networks
docker network create frontend-net
docker network create backend-net

# Run database on backend only
docker run -d --name db --network backend-net postgres

# Run app on both networks
docker run -d --name app --network frontend-net myapp
docker network connect backend-net app

# App can reach frontend and backend networks
```

## 7. DOCKER COMPOSE NETWORKING

```yaml
version: '3.8'

networks:
  frontend:
  backend:

services:
  database:
    image: postgres
    networks:
      - backend
  
  api:
    image: myapi
    networks:
      - frontend
      - backend
  
  web:
    image: nginx
    networks:
      - frontend
    ports:
      - "80:80"
```

## 8. BEST PRACTICES

- ✅ **Use custom networks** (not default bridge)
- ✅ **Plan IP addressing** (avoid conflicts)
- ✅ **Use meaningful network names**
- ✅ **Implement least privilege** (only necessary connections)
- ✅ **Document network topology**
- ✅ **Monitor network performance**
