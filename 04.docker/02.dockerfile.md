# Dockerfile Instructions (Definition, Purpose, Characteristics)

Dockerfile instructions define how a Docker image is built. Each instruction creates a new **image layer** during the build process.

---

# 1. FROM

## Definition
`FROM` specifies the **base image** from which the Docker image will be built.

## Purpose
- To define the starting point of the Docker image.
- Provides the operating system and basic environment.

## Characteristics
- It must be the **first instruction** in a Dockerfile.
- Can reference images from **Docker Hub or private registries**.
- Supports multiple stages (multi-stage builds).

## Example
```dockerfile
FROM ubuntu:22.04
```
---
<br>

# 2. RUN

## Definition

`RUN` executes commands during the **image build process** and commits the result as a new layer.

## Purpose

* Install packages
* Configure software
* Prepare environment for the application

## Characteristics

* Executes only during **image build time**.
* Each RUN instruction creates a **new layer**.
* Multiple commands can be chained using `&&`.

## Example

```dockerfile
RUN apt-get update && apt-get install -y nginx  # good practice
```

---
<br>

# 3. CMD

## Definition

`CMD` specifies the **default command** that runs when the container starts.

## Purpose

* Provides default execution behavior for the container.

## Characteristics

* Only **one CMD instruction** should exist.
* Can be overridden when running the container.
* Works together with `ENTRYPOINT`.

## Example

```dockerfile
CMD ["python", "app.py"]
```
---
<br>

# 4. EXPOSE

## Definition

`EXPOSE` declares the network port on which the container listens.

## Purpose

* Documents which ports the container uses.

## Characteristics

* Does **not publish the port** automatically.
* Used mainly for documentation and container networking.
* Port must be mapped using `docker run -p`.

## Example

```dockerfile
EXPOSE 80
```

---
<br>

# 5. ENV

## Definition

`ENV` defines environment variables inside the container.

## Purpose

* Configure application environment
* Store configuration variables

## Characteristics

* Variables are available during **build and runtime**.
* Can define multiple variables.

## Example

```dockerfile
ENV APP_ENV=production
```

---
<br>

# 6. ADD

## Definition

`ADD` copies files, directories, or remote URLs from the host system into the container.

## Purpose

* Transfer files into the container during image build.

## Characteristics

* Can extract **compressed tar files automatically**.
* Supports copying from **URLs**.
* May invalidate build cache.

## Example

```dockerfile
ADD app.tar.gz /app/
```

---
<br>

# 7. COPY

## Definition

`COPY` copies files or directories from the host machine to the container filesystem.

## Purpose

* Transfer application code or resources into the container.

## Characteristics

* Simpler and safer than ADD.
* Does not support URL downloads or archive extraction.

## Example

```dockerfile
COPY app.py /app/
```

---
<br>

# 8. ENTRYPOINT

## Definition

`ENTRYPOINT` defines the **main executable** that runs when the container starts.

## Purpose

* Ensure a container always runs a specific program.

## Characteristics

* Cannot be easily overridden.
* Often used together with CMD.

## Example

```dockerfile
ENTRYPOINT ["python"]
CMD ["app.py"]
```

---
<br>

# 9. VOLUME

## Definition

`VOLUME` creates a mount point for external storage.

## Purpose

* Store persistent data outside the container.

## Characteristics

* Data persists even if container is deleted.
* Used for databases and logs.

## Example

```dockerfile
VOLUME ["/data"]
```

---
<br>

# 10. USER

## Definition

`USER` sets the user identity used to run container commands.

## Purpose

* Improve security by avoiding root execution.

## Characteristics

* Applies to RUN, CMD, and ENTRYPOINT instructions.
* Can specify user or UID.

## Example

```dockerfile
USER nginx
```

---
<br>

# 11. WORKDIR

## Definition

`WORKDIR` sets the working directory inside the container.

## Purpose

* Define the directory where commands will execute.

## Characteristics

* Automatically creates directory if not present.
* Affects RUN, CMD, COPY, and ADD instructions.

## Example

```dockerfile
WORKDIR /app
```

---
<br>

# 12. ARG

## Definition

`ARG` defines variables that are available **only during the image build process**.

## Purpose

* Pass configuration values at build time.

## Characteristics

* Not available after container starts.
* Used with `docker build --build-arg`.

## Example

```dockerfile
ARG VERSION=1.0
```

---
<br>

# 13. ONBUILD

## Definition

`ONBUILD` adds a trigger instruction executed when the image is used as a base image.

## Purpose

* Automate actions in child builds.

## Characteristics

* Used for creating reusable base images.
* Triggered only when the image is used as a parent image.

## Example

```dockerfile
ONBUILD COPY . /app
```

---
<br>

# 14. STOPSIGNAL

## Definition

`STOPSIGNAL` defines the signal used to stop a container.

## Purpose

* Ensure graceful shutdown of applications.

## Characteristics

* Defines the system signal sent when stopping a container.

## Example

```dockerfile
STOPSIGNAL SIGTERM
```

---
<br>

# 15. LABEL

## Definition

`LABEL` adds metadata information to Docker images.

## Purpose

* Provide descriptive information about images.

## Characteristics

* Stores key-value pairs.
* Useful for automation and management tools.

## Example

```dockerfile
LABEL maintainer="nitin@example.com"
LABEL version="1.0"
```

---
<br>

# Difference Between COPY and ADD

| Feature              | COPY | ADD |
| -------------------- | ---- | --- |
| Copy files           | ✔    | ✔   |
| Copy directories     | ✔    | ✔   |
| Download remote URLs | ❌    | ✔   |
| Extract archives     | ❌    | ✔   |

Best Practice:
Use **COPY** for simple file transfers and **ADD** only when archive extraction or URL download is required.
