## 1️⃣ Maintainability

**Definition:**
Maintainability means software is **easy to understand, modify, fix, and extend** without breaking existing features.

**Example:**

```python
def add(a, b):
    return a + b
```

Using functions and proper structure makes the code easier to maintain.

**Real-life scenario:**
Companies like Amazon maintain huge codebases. Developers must update features (like payment or delivery logic) without breaking the system.

---

## 2️⃣ Continuous Delivery (CD)

**Definition:**
Continuous Delivery is a practice where code is **automatically built, tested, and prepared for deployment** whenever developers push changes.

**Example pipeline:**

```
Code Push → Build → Test → Deploy
```

Tools used:

* Jenkins
* GitHub Actions

**Real-life scenario:**
At Netflix, developers push code and automated pipelines test and deploy it many times per day.

---

## 3️⃣ Automation in Testing

**Definition:**
Automation means using tools to **run tests automatically instead of manual testing**.

**Example:**

```python
def test_add():
    assert 2 + 2 == 4
```

Tool used: pytest

**Real-life scenario:**
Banking apps run **thousands of automated tests** after every code change to ensure transactions work correctly.

---

## 4️⃣ Quality Gates

**Definition:**
Quality gates are **checks in a CI/CD pipeline** that ensure code meets certain standards before deployment.

**Example checks:**

* code formatting
* lint errors
* test results

If any fails → deployment stops.

**Real-life scenario:**
Companies like Google use strict quality gates so only high-quality code reaches production.

---

## 5️⃣ Test Pyramid

**Definition:**
The Test Pyramid is a testing strategy with **many unit tests, fewer integration tests, and very few UI tests**.

Structure:

```
UI Tests (few)
Integration Tests
Unit Tests (many)
```

**Example:**

* Unit: test a function
* Integration: API + database
* UI: test login page

Tools: Selenium

**Real-life scenario:**
E-commerce platforms like Flipkart test cart logic with unit tests and checkout flow with UI tests.

---

## 6️⃣ Code Quality Tools

Tools that check code quality automatically.

Examples:

* Flake8 → code linting
* Black → auto formatting
* MyPy → type checking

**Real-life scenario:**
Before merging code, CI pipelines automatically run these tools to ensure clean code.

# Mocking in Python

## What is Mocking?
**Mocking** is a testing technique where a **fake object (mock object)** replaces a real object during testing.  
It helps simulate the behavior of external systems like **APIs, databases, or services**.

## Why Mocking is Used
- To **avoid calling real external services**
- To make tests **faster and reliable**
- To **control responses** during testing
- To simulate **success or failure scenarios**

Example: If your code calls an external API, the API might be **slow, down, or return different data**. Mocking avoids this problem.

---

# Mocking Example in Python

## Function Code

```python
import requests

HEADERS = {}

def get_merge_requests(state=None):
    response = requests.get(
        "https://git.epam.com/api/v4/projects/124721/merge_requests",
        params={"state": state, "per_page": "100"},
        headers=HEADERS
    )

    res = []
    for item in response.json():
        res.append({
            "title": item["title"],
            "num": item["iid"],
            "link": item["web_url"]
        })

    return res
```

# Nose + Coverage (Python Testing)

## 1. Install Tools

```bash
pip install nose
pip install coverage
````

* **nose** → A Python testing framework used to discover and run tests.
* **coverage** → Measures how much of your code is executed during tests.

---

# 2. Run Tests

```bash
nosetests
```

This command:

* Automatically finds test files (like `test_*.py`)
* Executes all tests in the project.

---

# 3. Run Tests with Coverage

```bash
nosetests --with-coverage --cover-package=handlers/ --cover-erase
```

### Meaning of each option

| Option                      | Purpose                                            |
| --------------------------- | -------------------------------------------------- |
| `--with-coverage`           | Enables code coverage measurement                  |
| `--cover-package=handlers/` | Measures coverage only for the `handlers` package  |
| `--cover-erase`             | Clears previous coverage data before running tests |

This tells you **what percentage of your code is tested**.

Example output:

```
Name                 Stmts   Miss  Cover
----------------------------------------
handlers/file.py       50      5    90%
```

---

# 4. Enforce Minimum Coverage

```bash
nosetests --with-coverage --cover-package=handlers/ --cover-erase --cover-min-percentage=90
```

This ensures:

* Tests must cover **at least 90% of the code**.
* If coverage is below 90%, the test **fails**.

Example:

```
Coverage failure: total of 85 is less than fail-under=90
```

This is often used in **CI/CD pipelines**.

---

# 5. Generate HTML Coverage Report

```bash
nosetests --with-coverage --cover-package=handlers/ --cover-erase --cover-html
```

This generates a **visual coverage report**.

Report location:

```
cover/index.html
```

Open it in a browser to see:

* which lines were executed
* which lines were not tested

Example view:

```
Green → tested lines
Red   → untested lines
```

---

# 6. Testing Process Flow

```
Write Code
     ↓
Write Unit Tests
     ↓
Run Tests (nosetests)
     ↓
Measure Coverage
     ↓
Check Coverage Threshold
     ↓
Generate Coverage Report
```

---

# Key Idea

**Code Coverage** measures how much of your code is executed by tests.

Example:

```
Total code lines: 100
Tested lines: 90

Coverage = 90%
```

Higher coverage → **better test reliability**.

# Pytest 

## What is Pytest?
**Pytest** is a popular Python testing framework used to **write and run tests easily**.  
It supports **simple syntax, powerful fixtures, and plugins**.

---

## Install Pytest

```bash
pip install pytest
```

---

Example file: `test_math.py`

```python
def add(a, b):
    return a + b

    assert add(2, 3) == 5
```

---

## Running Tests

```bash
pytest
```

Pytest automatically:

* Finds test files (`test_*.py`)
* Executes test functions (`test_*`)

Example output:

```

```

---


# Tox

## What is Tox?
**Tox** automates **testing across multiple Python environments**.  
It creates virtual environments, installs dependencies, and runs tests automatically.

---

## Why Use Tox?
- Test your package on **different Python versions**
- Ensure correct installation of dependencies
- Simplify testing in **CI/CD pipelines**

---

## Install Tox
```bash
pip install tox
````

---

## Example: tox.ini

```ini id="k6xk7h"
[tox]
envlist = py38, py39, py310

[testenv]
deps = pytest
commands = pytest
```

* `envlist` → Python versions to test
* `deps` → dependencies for the test environment
* `commands` → test command to run

---

## Run Tox

```bash
tox
```

Tox will:

1. Create virtual environments for Python 3.8, 3.9, 3.10
2. Install `pytest` in each environment
3. Run `pytest` tests in all environments

---

## Output Example

```
py38: commands succeeded
py39: commands succeeded
py310: commands succeeded
```

# Packaging & Dockerizing Python Applications

## 1. Python Packaging Options
- **Native Python Packaging** → Wheel (`.whl`) or source archive (`.tar.gz`)  
- **System Packages** → RPM (RedHat) or DEB (Debian)  
- **Conda Packages** → For Conda environments  
- **Freezers** → PyInstaller, PyOxidize (create standalone executables)  
- **Container Images** → Docker images  
- **Virtual Machines** → Full OS images  
- **Hardware** → Direct deployment to devices (plug-and-play)

---

## 2. Dockerizing Python Applications

### Dockerfile Example

```dockerfile id="zv9m5k"
# Base Python image
FROM python:3.8

# Set working directory
WORKDIR /code

# Copy dependencies
COPY requirements.txt .

# Install dependencies
RUN pip install -r requirements.txt

# Copy source code
COPY src/ .

# Command to run application
CMD ["python", "./server.py"]
````

### Build & Run

```bash id="a5ln7d"
# Build the Docker image
docker build -t my-python-app .

# Run the container
docker run -it --rm --name my-running-app my-python-app
```

---

## 3. Docker Image Types

| Image Type        | Description                                                                      |
| ----------------- | -------------------------------------------------------------------------------- |
| **-slim**         | Minimal packages, smaller size, needs Unix knowledge                             |
| **-alpine**       | Very small Alpine Linux base, space-efficient, but may have compatibility issues |
| **Full official** | Full Python image based on Debian, safest choice, easy to start projects         |
